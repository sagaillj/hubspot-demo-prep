**B2. Item 2 — NPS form quality**
- *Plan schema extension:* form objects can now declare `theme` (color hex), `header_logo_url`, and field types `dropdown_select` (with `options: [{label, value}]`) and `multi_line_text`. Branded theme pulls primary color from `research["branding"]["primary_hex"]`.
- *Phase 2 synthesis:* when generating an NPS form, write:
  ```json
  {
    "name": "Post-Install NPS Survey (Demo)",
    "fields": [
      {"name": "firstname", "label": "First Name", "field_type": "single_line_text", "required": true},
      {"name": "lastname", "label": "Last Name", "field_type": "single_line_text", "required": true},
      {"name": "email", "label": "Email", "field_type": "email", "required": true},
      {"name": "nps_score", "label": "On a scale of 1-10, would you recommend {company} to a friend or colleague?", "field_type": "dropdown_select", "options": [{"label":"1","value":"1"},...,{"label":"10","value":"10"}], "required": true},
      {"name": "nps_feedback", "label": "Tell us about your experience", "field_type": "multi_line_text", "required": false}
    ],
    "submit_text": "Send Feedback",
    "theme": {"primary_hex": "{primary_hex}", "submit_button_color": "{primary_hex}"},
    "test_submissions": 6
  }
  ```
- *Builder change:* `create_forms` learns `dropdown_select` (HubSpot v3 fieldType `dropdown` with `options`) and `multi_line_text` (fieldType `multi_line_text`). Adds `displayOptions.style` block: `{ submitColor, submitFontColor, labelTextColor }` derived from theme. Form-group constraint (max 3 fields per group) preserved — 6 fields → 2 groups.
- *Test submissions:* generate realistic scores (mostly 8-10, some 5-7, a few 1-4) and 1-2-sentence feedback strings.

- DOD: rendered NPS form has the question + dropdown + textarea + branded submit button; sample submissions visible on contacts.

- Risk: medium-high (new field types, schema change). Don't break: existing 3-field forms (Marine Consultation, Remote Starter Quote) still work; v3 API max-3-fields-per-group constraint; form_submit endpoint behavior.

## Batch C (largest, new external integration)

**C1. Item 1 — Image-gen provider chooser**
- *Architecture:* new `helpers/09-generate-hero.sh` (orchestrator-side, runs in Phase 2 Claude session). Detects providers in priority order:
  1. **Recraft MCP** (preferred): if `mcp__recraft__generate_image` callable in this session, use it. Free tier = 30 credits/day.
  2. **OpenAI gpt-image-1**: if `OPENAI_API_KEY` set in env, curl `POST /v1/images/generations` with `model=gpt-image-1`, decode base64, save to `/tmp/demo-prep-<slug>/hero-image.png`.
  3. **Google Gemini**: if `GEMINI_API_KEY` (or `GOOGLE_AI_STUDIO_KEY`) set, curl `generativelanguage.googleapis.com/v1beta/models/imagen-3.0-generate-001:predict`.
  4. **Codex/ChatGPT environment**: if `CODEX_HOME` or `CHATGPT_*` env signals present, fall through to a manual_step pointing at the chooser CLI.
  5. **Skip with manual_step** if none available.
- *Preflight one-liner:* SKILL.md Phase 2 instructions tell Claude to print "Using Recraft (free tier). Want to switch?" before generating. User can override via env `HUBSPOT_DEMO_HERO_PROVIDER=openai|gemini|recraft`.
- *Prompt template:* "Hero image for a marketing email to {company} customers in {industry}. Brand colors: primary {primary_hex}, secondary {secondary_hex}. {pain_point_relevant_imagery}. Photorealistic, modern, no text overlay, 1200x600 aspect."
- *Builder side:* zero changes. `upload_hero_image()` already consumes `plan["marketing_email"]["hero_image_path"]`. The new helper writes that path.
- DOD: fresh-prospect run completes end-to-end with a generated hero showing in the marketing email; preflight surfaces the chosen provider; `--no-image` flag skips generation.
- Verification: render marketing email in HubSpot UI, screenshot, confirm hero image is non-default and visually relevant. Test with Recraft (real call), then mock OpenAI/Gemini with offline fixtures.
- Risk: medium. Don't break: existing flow when `hero_image_path` is pre-supplied (manual override).
